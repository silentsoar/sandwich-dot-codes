## Overview

Expecto is an interactive neural network visualizer that makes the inner workings of a neural network tangible and playful. You build a network by adding and removing layers and neurons, choose an activation function, pick a dataset, and hit Train. As the network learns, you watch connections thicken and color based on weight values, signal particles flow between neurons, and decision boundaries paint themselves across the canvas.

The core idea is that neural network toolkits are either too abstract (loss curves and tensor shapes) or too academic (static diagrams in textbooks). Expecto occupies the middle ground: a live, interactive sandbox where you can drag neurons around, click inputs to toggle them, and see exactly what each neuron is doing at any given moment. No install, no setup, no server — just open the page and start poking.

## The Problem It Solves

Most neural network educational tools fall into two categories. The first is the "tutorial notebook" — a Jupyter or Colab notebook that trains a model and plots a loss curve. These are informative but passive. You run the cell, you see the output, you don't interact. The second is the "textbook diagram" — a static illustration of circles and arrows with labels like "hidden layer 1." These give you the vocabulary but not the intuition.

What's missing is the middle layer: a tool where you can *feel* the weights changing, where backpropagation looks like something physical happening, where you can break a network and see *why* it breaks. Expecto fills that gap. It's a sandbox, not a tutorial. There are no right answers, no progress bars, no curriculum. You poke it and it squirms.

## Development Process

The app is built entirely in vanilla JavaScript with no build tools, no bundler, and no framework. TensorFlow.js handles the machine learning, and the HTML Canvas 2D API handles all rendering. The whole thing is three files: `index.html`, `style.css`, and `script.js`.

### Neural Network Engine

The `NeuralNetwork` class wraps a TensorFlow.js sequential model. The architecture is stored as a plain array (default `[2, 4, 4, 1]` — two inputs, two hidden layers of four neurons each, one output), and the model is rebuilt from scratch whenever the architecture changes.

```javascript
buildModel() {
  const layers = [];
  for (let i = 0; i < this.architecture.length - 1; i++) {
    const units = this.architecture[i + 1];
    const isLast = i === this.architecture.length - 2;
    layers.push(
      tf.layers.dense({
        inputShape: i === 0 ? [this.architecture[0]] : undefined,
        units,
        activation: isLast ? "sigmoid" : this.activation,
        kernelInitializer: "glorotUniform",
        biasInitializer: "zeros",
      })
    );
  }
  this.model = tf.sequential({ layers });
  this.model.compile({
    optimizer: tf.train.adam(0.01),
    loss: "binaryCrossentropy",
  });
}
```

Three activation functions are supported: tanh (default), ReLU, and sigmoid. The output layer always uses sigmoid since all three datasets are binary classification problems. The optimizer is Adam with a user-adjustable learning rate (logarithmic scale from 0.0001 to 0.1, default 0.01).

The architecture is mutable at runtime. Users can add or remove hidden layers and add or remove neurons from all hidden layers simultaneously. Each mutation disposes the old model, rebuilds, and re-extracts weights for visualization. The parameter count is recalculated and displayed in real-time.

### Training

Training runs in a `requestAnimationFrame` loop, not in a blocking loop. Each frame executes a configurable number of training steps (1–20, default 5) via `model.trainOnBatch()`. This keeps the UI responsive during training — the visualization continues to render at 60fps while the model learns.

Tensor disposal is critical here. Every `trainOnBatch` call returns a loss tensor that must be disposed manually. The input and label tensors are created once when training starts and disposed when training stops. The `getActivations` method wraps all intermediate tensors in `tf.tidy()` to prevent leaks.

The stats bar at the bottom of the screen shows live training status: whether the model is training or idle, the current epoch count, the latest loss value, and the total parameter count.

### Data Generation

The `DataGenerator` class provides three classic 2D classification problems:

- **XOR** — 200 random points in [-1, 1]² labeled by the XOR of their sign. A two-class problem that requires at least one hidden layer to solve.
- **Circle** — 200 random points labeled 1 if inside a circle of radius √0.5, 0 otherwise. A non-linear boundary that tests the network's ability to learn curved decision surfaces.
- **Spiral** — 300 points arranged in two interleaving spirals. The hardest of the three — requires a network with sufficient capacity and the right activation function to separate the classes.

Each dataset returns `{ data, labels, inputDim }` and is passed directly to TensorFlow.js tensors for training.

### Canvas Rendering

![Network visualization with decision boundary overlay and signal particles](/images/projects/expecto-network.png)

The `Renderer` class manages two stacked canvases: one for the network visualization and one for the decision boundary overlay. Both are rendered at device pixel ratio for sharp display on high-DPI screens.

The network canvas draws, in order:

1. **Grid** — A subtle dot grid at 40px intervals for visual texture.
2. **Connections** — Lines between every neuron in adjacent layers. Line thickness and opacity scale with the absolute weight value. Positive weights are teal (`#6f9d9a`), negative weights are salmon (`#d98b73`).
3. **Signals** — Animated particles that travel along connections. Each frame, random signal pairs are spawned between adjacent layers. The particle color matches the sign of the source neuron's activation. Signals accelerate as they travel (eased progress curve) and fade out at the midpoint.
4. **Neurons** — Circles filled with dark (`#292524`) and stroked with a color that indicates their role: mustard (`#d6b347`) for input neurons, teal for hidden neurons, salmon for output neurons. Stroke alpha scales with the absolute activation value, so neurons "light up" when they're active. Each neuron displays its activation value (or label for input/output neurons). A radial glow gradient behind each neuron provides additional visual feedback.
5. **Layer labels** — "Input", "Hidden 1", "Hidden 2", ..., "Output" positioned below each layer column.

### Interactivity

Neurons are draggable. Click and drag any neuron to reposition it. The renderer tracks the drag offset and constrains the neuron to the canvas bounds. On touch devices, touch events are handled separately from mouse events with the same drag logic.

Clicking an input neuron toggles its value between 0 and 1. This triggers a forward pass through the network with the new input, updates all neuron activations, and spawns a burst of signal particles to visualize the data flowing through the network.

A tooltip follows the cursor on hover, showing the neuron's layer, index, and current activation value (e.g., "L1·N2 | activation: +0.3842").

### Decision Boundary

The decision boundary is a separate canvas layer rendered at reduced resolution (60×60 grid). For each grid cell, the network predicts the output for the corresponding (x, y) coordinate in [-1, 1]². The cell is colored by interpolating between salmon (prediction = 0) and teal (prediction = 1) at 18% opacity, composited with screen blend mode.

The boundary updates every 10 training steps during active training. This is the most expensive operation in the app — 3,600 forward passes per update — which is why it's rendered at low resolution and throttled during training.

## Technology

The entire app runs client-side in the browser. No server, no backend, no build step. Three files, loaded directly.

### TensorFlow.js

TensorFlow.js (v4.17.0, loaded from CDN) provides the neural network primitives. The app uses `tf.sequential()` to build dense feed-forward networks, `model.trainOnBatch()` for single-batch gradient descent, and `model.predict()` for inference. Weights are extracted via `layer.getWeights()` after each training step for visualization.

Tensor management is the most error-prone part of the integration. The app creates tensors for training data once and disposes them on training stop. Intermediate tensors from `getActivations()` are wrapped in `tf.tidy()`. Loss tensors from `trainOnBatch()` are disposed immediately after reading their value.

### Canvas API

All rendering uses the Canvas 2D API. The network visualization canvas handles neuron circles, connection lines, signal particles, and layer labels. The boundary canvas is a separate element layered on top with CSS `mix-blend-mode: screen` and 30% opacity.

Canvas dimensions are scaled by `window.devicePixelRatio` for crisp rendering on retina displays. The canvas is resized on window resize events, and the neuron layout is recalculated to fill the available space.

Animation runs in a `requestAnimationFrame` loop that calls `renderer.render()` every frame, clearing and redrawing the entire network visualization. Signal particles are updated and pruned each frame — expired signals (progress ≥ 1) are filtered out.

### Drag and Drop

Neuron dragging is implemented with pointer math on `mousedown`, `mousemove`, and `mouseup` events. Touch equivalents (`touchstart`, `touchmove`, `touchend`) are handled separately with `preventDefault()` to suppress scroll. The drag offset is calculated from the click position relative to the neuron center, and the neuron position is clamped to canvas bounds during the drag.

The app distinguishes between a click and a drag using a `hasDragged` flag. If the mouse moves during a mousedown-mouseup cycle, the click handler is suppressed. This prevents accidental input toggles when the user intended to drag.

## UI Design

![Control panel with network architecture controls, training settings, and display toggles](/images/projects/expecto-controls.png)

The interface is a single-screen layout with no scrolling. A left sidebar (300px) contains all controls, organized into three collapsible sections: Network, Training, and Display. The main area is the canvas. A stats bar at the bottom shows live training metrics.

The visual language is warm and tactile. The color palette is built around dark warm greys (`#1a1816`, `#211e1b`, `#292524`) with four accent colors: mustard (`#d6b347`) for primary actions and input neurons, teal (`#6f9d9a`) for positive values and hidden neurons, salmon (`#d98b73`) for negative values and output neurons, and slime (`#9fb06f`) for active/training states.

Buttons have tactile box shadows (`4px 4px 0px 0px #463f3a`) that compress on click, giving a physical "press" feel. Panel sections have subtle rotations (`±0.2deg`) for a hand-assembled aesthetic. A noise texture overlay at 3.5% opacity adds grain across the entire interface. The header and control panel section headings use Fraunces, a variable optical size serif, while body text uses Inter.

A decorative squiggle SVG divider separates the header from the main content. Navigation links in the header rotate slightly on hover.

The UI is optimized for desktop but degrades gracefully on mobile: the control panel stacks above the canvas, nav links are hidden, and panel sections arrange horizontally with a minimum width of 220px.

## What I Learned

- TensorFlow.js is surprisingly capable for browser-based ML, but tensor disposal is a constant concern. Every `trainOnBatch`, `predict`, and `getActivations` call creates tensors that must be explicitly disposed or wrapped in `tf.tidy()`. Miss one and you have a memory leak that compounds over thousands of training steps.
- Canvas 2D is the right tool for this kind of visualization. WebGL would be faster for thousands of neurons, but at the scale of a toy neural network (2–12 neurons per layer), Canvas 2D is plenty fast and much simpler to work with. No shaders, no buffers, no coordinate transforms — just `beginPath`, `arc`, `stroke`.
- The decision boundary is the most expensive thing to render. 3,600 forward passes at 60×60 resolution, each running through the full TensorFlow.js model. Throttling it to every 10 training steps and rendering at low resolution keeps it interactive, but it's still the bottleneck.
- Making neurons draggable sounds trivial until you need to handle touch events, distinguish clicks from drags, and constrain positions to canvas bounds. The `hasDragged` flag pattern (set on first mousemove, checked in click handler) is simple but essential.
- The warm, tactile UI style — tactile shadows, noise overlays, slight rotations, serif headings — works well for a tool that's meant to feel approachable rather than clinical. It makes a neural network sandbox feel like a toy, which is the point.

## Challenges

- **Tensor lifecycle**: Managing TensorFlow.js tensor disposal across training loops, forward passes, and architecture changes required careful `tf.tidy()` wrapping and explicit `dispose()` calls. A single missed disposal during `getActivations` would leak hundreds of tensors per frame.
- **Decision boundary performance**: Rendering 3,600 predictions per boundary update is expensive. The solution was low-resolution grid (60×60), throttled updates (every 10 training steps), and a separate canvas layer with screen blend mode to avoid recompositing the network visualization.
- **Architecture mutations**: Adding or removing layers or neurons requires disposing the old model, rebuilding from the new architecture array, re-extracting weights, and relaying out neurons — all while preserving the positions of neurons that still exist. The `existingPositions` map passed to `layoutNetwork` handles this by matching neurons by their `layer-index` key.
- **Touch vs. mouse**: Mobile touch events don't map cleanly to mouse events. `touchstart` doesn't fire a `click`, `touchmove` doesn't have `clientX` directly (it's on `e.touches[0]`), and `preventDefault()` is needed to suppress scroll during drag. The app handles both input paths separately.
- **60fps rendering loop**: Drawing connections (N×M lines per layer pair), signal particles, neuron glows, and layer labels every frame at 60fps requires keeping the draw calls simple. No canvas shadows (too expensive), no gradients on connections (only on neurons), and signal particles are filtered out immediately when they expire.
